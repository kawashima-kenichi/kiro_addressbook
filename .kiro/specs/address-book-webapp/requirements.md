# Requirements Document

## Introduction

住所録WEBアプリケーションは、個人や組織の連絡先情報を効率的に管理するためのWEBベースのシステムです。ユーザーは連絡先の追加、編集、削除を行うことができ、データの永続化とセキュアなアクセス管理を提供します。

## Glossary

- **Address_Book_System**: 連絡先情報を管理するWEBアプリケーションシステム
- **Contact**: 個人または組織の連絡先情報（名前、住所、電話番号）
- **User**: システムを利用する認証済みのユーザー
- **Contact_Database**: 連絡先情報を永続化するデータベース
- **Authentication_System**: ユーザー認証を管理するシステム
- **Contact_Form**: 連絡先情報を入力・編集するためのフォーム
- **Contact_List**: 連絡先の一覧を表示するコンポーネント
- **User_Validator**: ユーザー登録時の入力データを検証するコンポーネント

## Requirements

### 要件1: ユーザー認証

**ユーザーストーリー:** ユーザーとして、個人の住所録にアクセスするために、システムに安全にログインしたい。

#### 受入基準

1. ユーザーが有効な認証情報（ユーザー名/メールアドレスとパスワード）を提供した場合、認証システムはユーザーを認証し、アクセスを許可する
2. ユーザーが無効な認証情報を提供した場合、認証システムはログイン試行を拒否し、汎用エラーメッセージ「ユーザー名またはパスワードが無効です」を表示する
3. ユーザーが15分以内に5回認証に失敗した場合、認証システムはアカウントを30分間ロックする
4. 認証システムは最後のアクティビティから正確に24時間、ユーザーセッションを維持する
5. 24時間の非アクティブ状態後にユーザーセッションが期限切れになった場合、認証システムは自動的にユーザーをログアウトし、ログインページにリダイレクトする
6. ユーザーがログアウトした場合、認証システムは即座にセッションを終了し、すべてのセッションデータをクリアする

### 要件2: 連絡先の追加

**ユーザーストーリー:** ユーザーとして、重要な連絡先情報を保存するために、住所録に新しい連絡先を追加したい。

#### 受入基準

1. ユーザーが有効な名前（1-100文字）を含む連絡先フォームを送信した場合、住所録システムは連絡先を連絡先データベースに保存する
2. 連絡先フォームは正確に1つのフィールドを必須とする：名前（最小1文字、最大100文字）
3. 連絡先フォームは任意フィールドを受け入れる：住所（最大500文字）、電話番号（最大20文字）
4. ユーザーがそのユーザーに既に存在する名前で連絡先を作成しようとした場合、住所録システムは送信を拒否し、エラー「この名前の連絡先は既に存在します」を表示する
5. ユーザーが無効なデータ（空の名前またはフィールド長の超過）を含む連絡先フォームを送信した場合、住所録システムは保存せずに具体的な検証エラーを表示する
6. 連絡先が正常に追加された場合、住所録システムは確認メッセージ「連絡先[名前]が正常に追加されました」を表示し、連絡先リストにリダイレクトする
7. 連絡先作成中にデータベースエラーが発生した場合、住所録システムはエラー「連絡先を保存できません。もう一度お試しください。」を表示し、フォームデータを保持する
8. 連絡先フォームは以下の形式の電話番号を受け入れる：(123) 456-7890、123-456-7890、123.456.7890、+1-123-456-7890、または1234567890

### 要件3: 連絡先の表示と一覧

**ユーザーストーリー:** ユーザーとして、住所録を閲覧するために、連絡先をリスト形式で表示したい。

#### 受入基準

1. 連絡先リストは認証されたユーザーに属するすべての連絡先を、名前のアルファベット順（大文字小文字を区別しない）で表示する
2. 連絡先リストは各エントリについて、連絡先名、電話番号（空の場合は「電話番号なし」）を表示する
3. 50件を超える連絡先が存在する場合、連絡先リストはページネーション制御（前へ/次へボタンとページ番号）付きで1ページあたり50件の連絡先を表示する
4. ユーザーが連絡先エントリをクリックした場合、住所録システムは詳細ビューで完全な連絡先詳細を表示する
5. ユーザーが連絡先を持たない場合、連絡先リストは「連絡先が見つかりません。最初の連絡先を追加して始めましょう。」というメッセージを表示する
6. 連絡先データベースが利用できない場合、連絡先リストは「連絡先を読み込めません。後でもう一度お試しください。」というエラーメッセージを表示する
7. 連絡先リストは最大10,000件の連絡先に対して読み込みと表示を行う
8. 連絡先リストは「Y件中X件を表示」という連絡先数インジケーターを表示する（Xは現在のページ範囲、Yは合計数）

### 要件4: 連絡先の編集

**ユーザーストーリー:** ユーザーとして、連絡先情報を最新の状態に保つために、既存の連絡先を編集したい。

#### 受入基準

1. ユーザーが特定の連絡先の「編集」ボタンまたはリンクをクリックした場合、住所録システムは既存のすべての連絡先データで連絡先フォームを入力する
2. ユーザーが有効な変更（名前1-100文字、電話番号は受け入れられる形式、住所最大500文字）を送信した場合、住所録システムは連絡先データベースの連絡先を更新する
3. ユーザーが無効な変更を送信した場合、住所録システムはデータを保存せずに具体的なフィールド検証エラーを表示する
4. 連絡先が正常に更新された場合、住所録システムは確認メッセージ「連絡先[名前]が正常に更新されました」を表示し、連絡先詳細ビューに戻る
5. ユーザーが編集中に「キャンセル」をクリックした場合、住所録システムはすべての変更を破棄し、保存せずに前のビューに戻る
6. 更新中にデータベースエラーが発生した場合、住所録システムはエラー「連絡先を更新できません。もう一度お試しください。」を表示し、再試行のためにフォームデータを保持する

### 要件5: 連絡先の削除

**ユーザーストーリー:** ユーザーとして、住所録を整理するために、不要になった連絡先を削除したい。

#### 受入基準

1. ユーザーが特定の連絡先の「削除」ボタンまたはリンクをクリックした場合、住所録システムは「[連絡先名]を削除してもよろしいですか？この操作は元に戻せません。」というメッセージの確認ダイアログを表示する
2. ユーザーが削除ダイアログで「確認」または「はい」をクリックした場合、住所録システムは連絡先データベースから連絡先を削除する
3. ユーザーが削除ダイアログで「キャンセル」または「いいえ」をクリックした場合、住所録システムはダイアログを閉じ、変更なしで連絡先を保持する
4. 連絡先が正常に削除された場合、住所録システムは確認メッセージ「連絡先[名前]が正常に削除されました」を表示し、連絡先リストを更新する
5. 削除中にデータベースエラーが発生した場合、住所録システムはエラーメッセージ「連絡先を削除できません。もう一度お試しください。」を表示し、連絡先を保持する
6. 確認ダイアログは30秒間の非アクティブ状態後に連絡先を削除せずに自動的に閉じる
7. 住所録システムは連絡先の削除を実行する前に、ダイアログを通じた明示的なユーザー確認を要求する

### 要件6: データの永続化

**ユーザーストーリー:** ユーザーとして、重要な情報を失わないように、連絡先データを安全に保存したい。

#### 受入基準

1. 連絡先データベースはユーザーアクションによって明示的に削除されるまで、すべての連絡先情報（名前、住所、電話番号、作成日、最終更新日）を永続的に保持する
2. システムが予期しないシャットダウンを経験した場合、連絡先データベースはすべてのコミットされたトランザクションが保持され、部分的な書き込みが存在しないことを確保してデータ整合性を維持する
3. 連絡先データベースはデータ破損なしに最大100人のユーザーによる同時アクセスをサポートする
4. 連絡先データベースは現地時間午前2時に24時間ごとに自動バックアップを作成し、30日間バックアップを保持する
5. データベース操作が失敗した場合、住所録システムはタイムスタンプとユーザーIDでエラーをログに記録し、ユーザーフレンドリーなメッセージ「システムエラーが発生しました。もう一度お試しください。」を表示する
6. バックアップシステムは週次でテスト復元操作を実行してバックアップの整合性を検証する
7. 連絡先データベースは通常の負荷条件下で、すべての書き込み操作（作成、更新、削除）を適切に完了する

### 要件7: レスポンシブデザイン

**ユーザーストーリー:** ユーザーとして、デスクトップとモバイルで連絡先を管理するために、異なるデバイスから住所録にアクセスしたい。

#### 受入基準

1. 住所録システムはデスクトップ画面（1024px以上）で水平スクロールなしにすべてのコンテンツを表示する
2. 住所録システムはサポートされるすべての画面サイズでCRUD操作（作成、読み取り、更新、削除）をすべて提供する

### 要件8: データ検証

**ユーザーストーリー:** ユーザーとして、データの品質と一貫性を確保するために、システムに入力を検証してもらいたい。

#### 受入基準

1. ユーザーが電話番号を入力した場合、住所録システムは以下の形式を受け入れる：(123) 456-7890、123-456-7890、123.456.7890、+1-123-456-7890、1234567890、その他の形式は「有効な電話番号を入力してください」というメッセージで拒否する
2. 住所録システムは名前（大文字小文字を区別しない）を比較して重複する連絡先を防止し、重複が検出された場合は「この名前の連絡先は既に存在します」というエラーを表示する
3. 住所録システムはフィールド長を制限する：名前（100文字）、住所（500文字）、電話番号（20文字）、制限を超えた場合は具体的なエラーメッセージを表示する
4. 任意のフィールドで検証が失敗した場合、住所録システムは有効なフィールドデータをクリアせずに、各無効フィールドの下にフィールド固有のエラーメッセージを表示する

### 要件9: セキュリティとプライバシー

**ユーザーストーリー:** ユーザーとして、機密情報が保護されるように、連絡先データを安全でプライベートにしたい。

#### 受入基準

1. 住所録システムは読み取り、更新、または削除操作の前に、ユーザーIDと連絡先の所有権を検証することで、ユーザーが自分の連絡先のみにアクセスできることを確保する
2. 住所録システムはクライアントとサーバー間のすべてのデータ送信にHTTPS（TLS 1.2以上）を使用する
3. 認証システムは保存前に最低12ラウンドのbcryptを使用してユーザーパスワードをハッシュ化し、平文パスワードを保存しない
4. 住所録システムはタイムスタンプ、ユーザーID、アクション種別、IPアドレスを含むすべてのデータアクセスと変更試行を監査証跡にログ記録する
5. ユーザーアカウントが削除された場合、住所録システムは24時間以内に関連するすべての連絡先データを削除する
6. 不正アクセス試行が検出された場合、住所録システムはアクセスを拒否し、セキュリティ違反をログに記録する
7. 住所録システムは24時間の非アクティブ状態後にセッションタイムアウトを実装し、再認証を要求する

### 要件10: ユーザー登録

**ユーザーストーリー:** 新規ユーザーとして、住所録システムを利用するために、アカウントを作成したい。

#### 受入基準

1. ユーザーが有効なメールアドレスと強力なパスワードを提供した場合、住所録システムは新しいアカウントを作成する
2. THE User_Validator SHALL パスワードが8文字以上で、大文字、小文字、数字、特殊文字を含むことを検証する
3. ユーザーが既に存在するメールアドレスで登録を試行した場合、住所録システムは登録を拒否し、「このメールアドレスは既に使用されています」エラーを表示する
4. 登録が正常に完了した場合、住所録システムは確認メッセージ「アカウントが正常に作成されました」を表示し、ログインページにリダイレクトする
5. IF 登録中にデータベースエラーが発生した場合、THEN 住所録システムは「アカウントを作成できません。もう一度お試しください。」エラーを表示し、フォームデータを保持する
6. ユーザーが無効なメールアドレス形式を入力した場合、住所録システムは「有効なメールアドレスを入力してください」エラーを表示する
7. IF パスワード要件を満たさない場合、THEN 住所録システムは「パスワードは8文字以上で、大文字、小文字、数字、特殊文字を含む必要があります」エラーを表示する
